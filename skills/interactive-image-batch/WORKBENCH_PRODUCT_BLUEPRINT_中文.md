# DAOGE 工作台产品化蓝图

## 终版结论

截至当前版本，DAOGE 的产品化目标已经从“多页面技能集合”收口为 **单一主链、状态驱动、对话同步的可视化工作台**。

普通用户默认只走这一条链：

1. `references/examples/examples_catalog.html`
2. `task_center.html`
3. `workspace/workspace_home.html`
4. `workspace/prepare_workspace.html`
5. `workspace/result_workspace.html`
6. `workspace/exception_workspace.html`
7. `workspace/run_record.html`

状态源职责已经定稿：

- `workspace_live_state.json`：任务内实时状态源
- `workspace_state.json`：统一工作台状态模型
- `runtime_state.json`：运行阶段进度与执行状态
- `task_center_live_state.json`：跨任务总控实时状态源
- `workbench_state.json`：内部派生工作台快照，不是新的主状态源

页面生成策略已经定稿：

- 主链页默认生成并进入 `workspace/` 镜像入口。
- `review_board.html`、`rerun_board.html`、`run_overview.html`、`completion_board.html` 只属于 `result-details` / `all` 深看层。
- `prompt_preview.html`、`preflight_board.html`、`assets_board.html` 只属于 `prepare-details` / `all` 深看层。
- `storyboard_board.html` 只在分镜任务或结果深看场景出现。
- `daoge_portal.html` 与 `result_hub.html` 不再是产品入口；渲染器已移除，同名残留只作为运行时清理目标。
- `workspace/` 是正式用户入口；`internal/` 和 `debug/` 分别服务程序状态与维护诊断。

本文以当前新生架构为准；后续章节只保留仍能指导落地的产品规则。

## 一、这份蓝图解决什么问题

当前 `interactive-image-batch` 已经形成“单一工作台主链”的产品骨架。

已经完成的重点包括：

- 以 `workspace_state.json` 为核心的统一状态层
- `workspace_assets.json` / `workspace_timeline.json` 的配套资产层与时间线层
- 四个主工作台页面成为主链：
  - `workspace_home.html`
  - `prepare_workspace.html`
  - `result_workspace.html`
  - `exception_workspace.html`
- 页面说明语言开始进入统一中文语言系统

但是，这还不等于“真正的产品化工作台”。

当前距离最终目标还有三段路：

1. 语言治理已经成形，但页面治理仍未完全独立
2. 用户能看到的页面已经收敛，但“页面分级 / 产物分级 / 显示优先级”还缺乏统一快照层
3. 这套系统还更像“生成了一些 HTML 页面”，而不是“给普通用户的一整套可视化工作流 UI”

所以，下一大阶段的目标，不再是继续零散修页面，而是：

**把 DAOGE 从“多页面技能集合”推进为“单一主链、状态驱动、用户导向的可视化工作台产品”。**

## 当前总判断

截至当前版本，这份蓝图以“当前正式阶段”来理解，后续编号只作为执行切刀。

当前已经完成并基本稳定的部分：

- 单一主链已经成立
- 四个主工作台已经成为默认入口
- 中文优先的页面语言已经建立第一轮统一
- 首屏主控、阶段确认、对话协同、阶段接力已经完成第一轮收束
- 用户侧默认不再需要理解内部产物和按需补充页

当前正式进入的下一个大阶段是：

**页面治理层独立化。**

这一阶段的目标不是继续零散修文案，也不是继续加页面，而是把“页面该怎么显示、默认先看什么、哪些内容退到补充层、哪些内容只保留给专业层”变成一套独立治理规则。

也就是说，从这一轮开始，四个主页面要越来越像：

- 同一套工作台系统
- 同一套阅读密度规则
- 同一套对话协同语言

而不是四张风格相近、但仍然各自定义显示规则的页面。

### 当前阶段新增落地原则

从 2026-05-23 开始，这个阶段不再只做“单页收文案”，而正式进入：

- `主链 section 契约注册表`
- `content 区声明层`
- `默认层 / 按需层 / 专业层` 的统一治理
- 页面治理层与状态层的正式对齐

也就是说，后续四张主链页允许出现哪些 section、这些 section 属于哪一层、默认给谁看，不再散落在各页渲染器里，而要进入统一注册表。
同时，四张主链页 `content` 区的出现顺序，也不再由页面层自己硬编码拼接，而要由状态层声明 `contentSections`，页面层按声明顺序渲染。

### 当前阶段最新推进点

截至 2026-05-23 当前这一轮，页面治理层独立化已经不再局限于四张主链页内部。

这意味着，后续推进不再把：

- `中文模板展示板`
- `任务总控`

视为“主链之外的临时跳板”，而是要把它们也逐步纳入同一套入口状态语言。

也就是说，新的正式方向是：

- 四张主链页继续统一
- 入口层也开始状态驱动化
- 用户从“选任务”到“进准备页”这段路径，不再像走另一套系统

当前已经开始落地的第一刀是：

- 为 `entry_state.json` 建立更完整的入口层状态结构
- 让 `examples_catalog.html` 优先消费这套入口状态
- 把“当前入口 / 推荐下一步 / 入口层主控”从页面内硬拼，推进到状态层提供

后续紧接着要继续推进的是：

- `task_center.html` 也逐步并入同一套入口层语言
- 入口层与任务内四站主链形成真正连续的一体化工作台

## 六、剩余任务总规划

为了避免后续开发重新回到“零散修页面”，后面的工作统一按下面四个大阶段推进。

### 大阶段 1：页面治理层独立化

这是当前已经正式进入、并且必须优先做完的大阶段。

目标：

- 把“默认先看什么、哪些内容退到补充层、哪些内容只给专业层”从页面里剥离出来
- 让四张主链页共享同一套阅读密度与分组语言
- 让状态层直接产出治理说明，而不是每张页面各自判断

本阶段要完成的事：

- 状态层正式产出页面治理说明
- 四张主链页优先消费治理说明
- `heroCards / cockpit / decision / summary / section groups` 继续统一来源
- 建立 `contentSections` 声明层，让内容区顺序由状态层直接驱动
- 页面层只保留最小渲染兜底
- 建立主链 `section` 契约注册表，统一四张主链页的 section 类型与层级归属

完成标志：

- 四张主链页的阅读分组规则来自同一套状态层数据
- 四张主链页的 `content` 区顺序来自同一套状态层声明
- 页面层基本不再自己拼“为什么要这么展示”
- 四张主链页的 section 分层规则可以从统一注册表中被解释和追踪

### 大阶段 2：主链状态驱动化

这是页面治理层之后的直接下一个阶段。

目标：

- 让四张主链页尽量只吃 `workspace_state.json`
- 页面渲染器从“半状态层 + 半页面逻辑”继续向“纯渲染壳”靠拢

本阶段要完成的事：

- `home / prepare / result / exception` 的剩余 fallback 继续下沉
- `route / workbench / guides / sections / summaries` 统一由状态层主导
- 页面脚本继续减重，避免四页各自保留一套业务判断

完成标志：

- 页面层主要只负责版式、组件拼装和极少数结构兜底
- 任务判断、人话摘要、导航策略主要都由状态层给出

### 大阶段 3：产物分级与生成收敛

目标：

- 让“生成什么页面”和“用户默认看到什么页面”彻底分开
- 继续压缩普通用户可见层，避免导航和重复跳转再次膨胀

本阶段要完成的事：

- 重新审视高级辅助页是否都还需要默认生成
- 让条件页、高级页、归档页的生成策略更可控
- 为 HTML、JSON、Markdown 建立更清晰的分级角色
- 清理退役入口文件，避免非主链产物重新膨胀

完成标志：

- 普通用户只会自然进入单一主链
- 高级页只在确实需要时出现
- 中间产物的角色边界更清晰

### 大阶段 4：工作流工作台产品化

目标：

- 让这套 skill 不再只是“批量生图脚本 + HTML 输出”
- 让它更像“和 Codex 对话协同的一整套可视化工作流 UI”

本阶段要完成的事：

- 把对话协同和页面协同进一步打通
- 把下一步提示、阶段接力、资产沉淀做成稳定产品语言
- 继续评估后端 JSON / Markdown 是否还能再整合
- 为后续“全功能可视化工作台”形态留出扩展点

完成标志：

- 用户在对话框里推进任务时，工作台能持续承接状态、建议和沉淀
- 页面、状态、运行产物不再像三套松散系统

## 三、最终产品形态

### 1. 用户看到的主链

普通用户默认只需要理解四站：

1. `工作台首页`
   - 我现在到哪一步
   - 当前是否有阻塞
   - 下一步去哪里

2. `准备工作台`
   - 任务方向是否对
   - 当前是否可以放行
   - 素材约束是否稳定

3. `结果工作台`
   - 当前哪些结果可用
   - 哪些还要复核
   - 当前最值得做的下一步是什么

4. `异常工作台`
   - 哪些问题会阻塞主链
   - 是否值得补跑
   - 处理完后如何回主链

### 2. 条件页

条件页不是默认导航链，而是按需出现：

- `Storyboard 整板页`
- `审阅看板`
- `补跑页`
- `任务档案`

它们只能作为：

- 补充页
- 条件页
- 高级页

不能重新长成第二套主链。

### 3. 内部产物

内部 JSON / Markdown 依旧保留，但角色变化为：

- 状态源
- 调试源
- 归档源
- 专业层输入源

而不是普通用户主入口。

- 不移除原有原始列表
- 先增加摘要层
- 页面优先读摘要层，原始列表继续兜底
- 保证现有主链生成链路和测试用例稳定

- 推荐原因

### 第三步

让它与现有：

- 行动建议
- 对话协同
- 阶段确认

形成上下关系，而不是重复关系。

### 第四步

跑 smoke 测试，确认：

- 页面生成不回退
- 旧断言仍成立
- 新摘要层已被覆盖

### 6D-2 第二刀

在 `6D-1` 的基础上，继续把“同一事实被多处重复表达”的问题往统一摘要层收一轮。

这一刀重点不是删字段，而是继续建立**单一事实来源**：

- 在 `workspace_state.json` 增加 `sourceSummary`
- 把结果计数、预览计数、参考素材计数收成统一摘要视图
- 把最近阶段变化的标题、摘要和最近事件快照对齐到时间线摘要层
- 让首页、结果页、异常页优先读取 `workspace_state.sourceSummary`

这一刀完成后，状态读取顺序应当变成：

1. 优先读取 `workspace_state.sourceSummary`
2. 其次读取 `workspace_assets.summary` / `workspace_timeline.summary`
3. 最后才回退到原始列表和旧 `counts`

原则保持不变：

- 不做激进删字段
- 不破坏现有 smoke 测试
- 页面层优先少改结构，多改数据来源

### 6D-3 第三刀

在 `sourceSummary` 已经建立后，再往前推进一层：

- 让首页、结果页、异常页优先直接消费 `workspace_state.views.*`
- 把页面里仍然本地拼装的：
  - 驾驶舱摘要
  - 判断说明
  - 摘要卡片
  继续往统一视图状态层收

这一刀的目标不是改视觉，而是继续让页面壳变薄：

1. 页面优先渲染 `views.*.cockpitSummary`
2. 页面优先渲染 `views.*.decision`
3. 页面优先渲染 `views.*.summary`
4. 本地 fallback 只保留最小兜底

这样后续如果还要继续统一中文语言、阅读顺序或信息密度，就不需要三页各改一次，而是继续改共享状态层即可。

### 6D-4 第四刀

继续把“页面不信任视图层、自己再拼一次”的残留点收掉。

这一步聚焦三类区块：

- `route`
- `workbench`
- `cockpitSummary`

推进原则：

1. 页面优先直接消费 `workspace_state.views.*.route`
2. 页面优先直接消费 `workspace_state.views.*.workbench`
3. 页面里的 fallback 只保留最低限度兜底，不再承担默认主逻辑

这样主页面会更像真正的统一工作台壳：

- 视图层负责给出结构化判断
- 页面层负责渲染
- 后续再做统一语言或导航收束时，修改面会继续缩小

  - 阶段信号

但是，距离最终的“全功能可视化工作流 UI”还有明显差距。

当前最大的缺口已经不再是：

- 页面长得像不像同一家产品

- 某张卡片的间距是不是更顺

而是：

**用户虽然已经能看懂页面，但还没有真正获得一套“持续陪跑的工作流主控系统”。**

也就是说，下一大阶段的目标应该从：

- 让页面更像工作台

切到：

- 让工作台更像“任务正在运行中的主控系统”

### 这个阶段要解决的核心问题

1. 用户切到任意主链页时，仍然缺少“一眼就能接住全局任务”的主控条
2. 当前阶段、放行状态、异常压力、推荐动作虽然都已存在，但还分散在多个区块里
3. 工作台仍然更像“已生成页面的集合”，还没有形成“当前任务正在被这套 UI 主控”的感觉
4. 对话框和工作台虽然已经协同，但还没有形成真正的“双轨并行工作流控制面”

### 阶段 4 的北极星目标

用户打开任意主链页时，第一屏不仅能看到解释和摘要，更能先看到一条统一的：

- 当前任务是谁
- 当前走到哪一阶段
- 当前状态是否可继续
- 当前异常压力有多大
- 当前推荐动作是什么

的**任务主控条**。

这条主控条必须：

- 在四个主链页完全同语法
- 用任务语言表达，而不是程序字段语言
- 让用户切页后无需重新理解当前局面
- 让对话框与工作台之间真正像同一套系统

## 十七、下一个大阶段：首屏判断产品化

阶段 4 的主控系统化推进到当前这一轮后，新的边界已经很清楚：

- 任务主控条已经建立
- 状态栈已经开始统一
- 行动建议与对话协同已经逐步下沉进统一状态快照

这意味着，下一个大阶段不该再停留在“把更多模块做得更像”，而要进一步解决一个更本质的问题：

**用户进入任意主链页时，系统能不能用同一套首屏判断语言，立刻接住当前任务。**

也就是说，新的阶段目标应该从：

- 让工作台像主控系统

继续推进到：

- 让工作台像“会判断、会接力、会提示下一步”的首屏控制面

### 这个阶段重点解决什么

1. 驾驶舱摘要虽然已经统一出现，但仍有部分判断文案散落在页面层
2. 阶段确认虽然已经稳定存在，但还没有完全成为统一快照层产物
3. 四个主链页的首屏判断能力还没有完全变成“同一套状态产品”，而更像“同一套结构 + 各页自己的句子”
4. 后续如果继续做更大的工作台改造，首屏判断层不先产品化，页面就还会继续长分叉

### 阶段 5 的北极星目标

用户切到任意主链页时，第一屏看到的不是页面解释集合，而是一套统一的：

- 当前局面
- 当前动作
- 当前阻塞
- 当前推荐回复
- 当前确认状态

的首屏判断控制面。

这套控制面必须：

- 在四个主链页同语法
- 尽可能由 `workspace_state.views.*` 直接提供可渲染成品
- 让页面层只负责渲染，不再负责大量补文案
- 为后续“真正的工作流 UI”继续清路

### 阶段 5 的第一刀

**阶段 5A：驾驶舱摘要与阶段确认快照化。**

这一刀的目标不是增加新区块，而是把已经被证明有价值的首屏判断层，继续从页面脚本里抽回统一视图状态层。

## 十八、下一个大阶段：任务会话操作系统化

在首屏判断层逐步稳定之后，下一阶段不能再只停留在“每一页更统一一些”。

因为对普通用户来说，真正的体验断点不是某一张卡片写得不够好，而是：

- 从首页跳到准备页后，像进入了另一套页面
- 从准备页跳到结果页后，又要重新理解当前局面
- 从结果页进入异常页后，缺少一个持续一致的“当前任务还在同一次会话里”的感觉

所以，新的大阶段目标不是再加更多页面，也不是再加更多说明区，而是：

**把四个主工作台进一步收束成同一场持续运行的任务会话。**

这意味着，用户无论落在哪一页，第一眼都应该先被同一块任务会话快照接住，而不是重新阅读一整页解释。

### 阶段 7 的北极星目标

用户在 Codex 对话框中继续和 DAOGE 交互时，四个主工作台要表现得像同一套持续运行的任务操作系统，而不是四张彼此相似、但仍然相互独立的 HTML 页面。

这套任务操作系统至少要持续回答六件事：

- 当前任务
- 当前阶段
- 当前状态
- 当前压力
- 现在动作
- 本轮规模

### 为什么要进入这个阶段

原因很直接：

1. 首屏判断已经开始统一，但用户跨页时的“会话连续感”还不够强
2. 页面虽然在收敛，但每一页仍然更像自己的页面，而不是同一套任务空间里的不同阶段视图
3. 如果未来要把这套工作台继续做成“对话框 + 工作台”的并行协同 UI，那么会话层必须先独立出来

也就是说，阶段 7 不是视觉装饰升级，而是主链工作台继续产品化的必要底盘。

### 阶段 7 的实施顺序

#### 7A：统一任务会话快照

目标：

- 在四个主链页首屏加入完全同语法的任务会话快照
- 让用户切页后仍然感觉自己停留在同一场任务会话里
- 把“当前任务 / 当前阶段 / 当前状态 / 当前压力 / 现在动作 / 本轮规模”收成固定锚点

原则：

- 这是任务会话快照，不是第二个摘要区
- 只说对人有用的任务语言，不说程序字段语言
- 页面层优先读取 `workspace_state.views.*.sessionConsole`
- 页面只负责渲染，不再临时拼第二套会话解释

#### 7B：跨页任务身份统一

目标：

- 让首页、准备页、结果页、异常页在任务称呼、阶段称呼、动作称呼上保持完全一致
- 进一步减少“同一件事在不同页面换一种说法”的情况

重点：

- 统一任务身份短句
- 统一阶段名与阶段职责短句
- 统一动作动词与回流语言

当前这一刀的具体落点应该是：

- 建立跨页统一的页面身份词典
- 建立跨页统一的阶段身份词典
- 建立跨页统一的主链流向词典
- 让状态层与页面 fallback 都优先读取这套词典，而不是继续在各页手写默认说法

继续往下一刀时，要进一步完成：

- 建立跨页统一的动作词典
- 建立跨页统一的回流动作词典
- 建立跨页统一的默认继续句词典
- 让驾驶舱摘要、主控判断、阶段信号、页面 fallback 全部优先读取这一套动作语言

#### 7C：阶段回流与下一步节奏统一

目标：

- 让“从上一站来 / 这一站做什么 / 下一站去哪”继续向统一节奏推进
- 让工作台与对话框的接力语言真正形成同一套工作流节奏

重点：

- 强化阶段接力的跨页连续感
- 让推荐回复、主动作、回主链提示使用同一套语言系统
- 为后续更大的工作流 UI 改造清理掉页面分裂感

这一刀落地时，应优先把这些区块收成同一条节奏链：

- 阶段接力
- 主控判断
- 驾驶舱摘要
- 阶段确认
- 对话协同

也就是说，用户在不同页面看到的“回到对话框怎么继续”，不该是五套并列说法，而应该是一条被不同区块复用的统一节奏对象。

进一步收口时，应优先保证这条统一节奏对象至少包含：

- 当前建议回复
- 主回复句
- 备选回复句
- 当前节奏摘要
- 可复用的接力短句

当这条统一节奏对象已经接入首屏核心区块后，下一步就不该继续加解释，而应该转入首屏减重：

- 任务主控条：只保留任务与阶段
- 任务会话快照：保留全局锚点
- 驾驶舱摘要：保留局面判断
- 阶段信号：退成轻量状态条

目标不是减少能力，而是减少同一信息在首屏被重复解释四次。

### 阶段 7 的第一刀

正式从这里开始，进入：

**阶段 7A：统一任务会话快照。**

这一刀的核心价值，不是让页面“看起来多一块内容”，而是让四个主工作台第一次真正像同一套持续运行的任务会话系统。

3. 页面渲染优先消费共享协同契约，而不是继续各页局部拼接

本轮第一刀的实际落点：

- 在 `workspace_state` 顶层新增统一协同注册层
  - 建议命名：
    - `workflowSessions`
    - `workflowCopilotRegistry`
- 这一层至少收纳：
  - `home`
  - `prepare`
  - `result`
  - `exception`
- 每一项继续复用当前已经稳定的 `workflowCopilot` 结构，不重新发明第二套 schema
- 页面层优先读取：
  - `workspaceState.workflowSessions[stage]`
  - 若缺失，再退回 `workspaceState.views[stage].workflowCopilot`

这样做的意义不是“多存一份数据”，而是把当前已经散落在四页 view 对象里的协同协议正式提升成：

**工作台共享协同底盘。**

这会让后续几刀更顺：

- 8B：协同提示减重
- 8C：对话框与工作台共享推荐回复对象
- 8D：阶段接力与确认统一注册
- 8E：页面 fallback 继续变薄
